# TASK_TEMPLATE — 任務單模板

使用方式：每次要交任務給 AI agent 時，複製下方模板、填空、連同
`AGENT_BOOT_PROMPT.md` 的內容一起貼給 agent。
空欄不要留白猜測——寫「無」也比留白好，留白會讓 agent 自行腦補。

---

```markdown
# 任務單：〔一句話標題〕

日期：YYYY-MM-DD

## 1. 本次任務目標

〔要達成什麼。用「完成後會看到什麼」描述，不要只寫動作。
 例：「後台 Theme 頁新增一個復原按鈕，按下後回到上次存檔前的規則」
 而不是「加一個按鈕」。〕

## 2. 可修改檔案

〔白名單。明確列路徑，可用資料夾＋萬用字元。
 例：
 - pages/admin/theme/assets/js/app.js
 - pages/admin/theme/assets/css/page-admin-theme.css
 - version.js（push 前 bump，固定允許）
 - AI_CONTEXT/PROJECT_STATE.md、AI_CONTEXT/CHANGELOG.md（交接更新，固定允許）〕

## 3. 禁止修改檔案

〔本次特別要保護的檔案。RULES.md 的通用禁令不用重抄，這裡只列本次額外的。
 例：assets/js/entry-core.js（本次任務不涉及水庫本體）
 沒有額外的就寫「僅 RULES.md 通用禁令」。〕

## 4. 已知前提

〔agent 不用重新驗證、直接採信的事實。
 例：「theme_css_rules 表已有約 40 條規則」「此 bug 只在手機寬度出現」。
 沒有就寫「無，一切以 AI_CONTEXT 文件為準」。〕

## 5. 驗收標準

〔本次任務專屬的驗收條件，逐條可勾選。通用驗收（ACCEPTANCE.md）不用重抄，
 agent 本來就要跑。例：
 - [ ] Theme 頁按復原後，畫面立即回到上次存檔狀態
 - [ ] 復原後重新整理，規則仍是復原後的內容
 - [ ] 外部專案引用的 CSS 變數同步回復〕

## 6. 回報格式

完成後依序回報：
1. 實際修改的檔案清單（與第 2 節白名單比對，有出入要說明原因）
2. 每個檔案改了什麼、為什麼
3. 驗收結果：第 5 節逐條 ✅/❌ + ACCEPTANCE.md 通用項逐條結果
4. 未完成/被擋住的部分與原因
5. 建議（不動手的那種）：發現的風險、值得後續處理的事
6. 是否已更新 PROJECT_STATE.md 與 CHANGELOG.md

## 7. 遇到衝突時的處理方式

- 任務要求與 RULES.md 衝突 → **停下來問**，附上衝突的具體條文，不要自行取捨。
- 任務要求與現有架構衝突（做了會破壞既有流程）→ 停下來說明衝突點與
  替代方案，等使用者裁決。
- 文件（AI_CONTEXT）與程式碼現況不符 → 以**程式碼**為準，回報文件過時之處，
  並在完成任務時順手修正文件。
- 需要修改白名單外的檔案才能完成 → 停下來申請，說明為什麼非改不可。
- 任何「不確定但影響方向」的問題 → 問，不要猜。小的實作細節可自行決定，
  但要在回報中列出你替使用者做了哪些決定。
```
