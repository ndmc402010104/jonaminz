# Minz Page v0.1 — 規格與 Agent 交接文件

**文件狀態：** 已完成第一輪產品定位與資訊架構決策，尚未實作。
**適用範圍：** `minz.jonaminz.com`（Minz 的房間，公開個人選品/內容展示

**核心原則：** Minz 頁面不是技術管理系統，而是 Minz 對生活內容的個人

原始來源：使用者準備的規格文件（`Minz_Page_v0.1_Agent_Handoff.md`），

處理方式。

## 0. 一句話定位

> **Minz 頁面是一個真實、帶有可辨識細節的個人生活選集，第一階段以旅行
> 內容為主，長期會逐步擴充成完整的個人生活頁。**

技術專案負責規劃、紀錄與整合真實使用者資料；Minz 頁面只展示已完成、
值得分享的精選內容。

## 1. 已確認的產品定位

### 1.1 第一階段與長期方向

**第一階段**：採用「技術為主」的骨架結構，因為這是目前最完整且明確的
內容領域，未來會用相同框架容納其他生活內容類別。

**長期方向**：逐步變成完整的 Minz 個人生活頁。技術內容只是入口內容，
不永久佔滿整個網站。

**結構要求**：目前的分類入口不是永久寫死的網站骨架，未來必須可以：

- 新增分類
- 改名
- 合併
- 調整排序
- 逐步淘汰
- 掃描過期

使用 **registry-driven navigation / section registry**，不可把分類
直接硬編碼進頁面骨架。

第一版沒有展示空間的分類，也不放「敬請期待」卡片，但結構不要預設未來
的擴充能力。

## 2. Minz 的頁面人格

### 2.1 核心氣質

> **生活感 × 內斂可愛**

多元且幼稚的美，也不會嚴肅到讓陌生訪客感到冷漠。可愛藏在細節、文字、
小圖示或小標籤裡。

### 2.2 人物形象

頁面該讓訪客感受到：

- 優雅
- 細膩
- 有自己的品味
- 喜歡懷舊歷史或深度旅行
- 收集小物件
- 對色調、質感與小事物有敏感度與偏好
- 不誇張，但有個人色彩

### 2.3 首頁名字

只使用：

```
Minz
```

不用中文全名，不使用 `MINZ`，也不加 `Travel Notes` 這種副標。

### 2.4 首頁短句

> **「歡迎光臨，旅行、好物與想聊的東西，還有一些可愛的小東西。」**

整體文字風格：幾句話不成篇章、生活化語氣。

### 2.5 About 文字方向

職業可以明確寫出，但不做履歷式呈現。

建議措辭：

> **「平日是一位營養師，休假時就會四處尋找美食與小模式。」**

職業資訊放在 About 區、作者調帶過，不出現在首頁主要視覺主焦點。

## 3. 首頁視覺與資訊架構

### 3.1 首頁形式

採用：

> **生活選集感**

首頁不是完整人物形象頁，也不是技術部落格首頁，而是 Minz 的個人入口與
內容選單。

### 3.2 形象與色彩

人物照片與旅行手帳，而不是完全以圖片為主視覺。

建議形式：

- 手繪風、沒有過度精緻感
- 手寫貼紙般手感
- 小圖示（郵戳/票券/地圖標籤/小圖章）
- 保留照片本身的乾淨真實感
- 裝飾不搶過人物
- 不要做成履歷大頭照片

### 3.3 主色調

> **米白 × 墨綠 × 淡棕**

可少量使用淡黃或淡粉作為點綴色。

視覺語彙：

- 米白：紙張、手帳、乾淨
- 墨綠：植物、旅行、沉靜
- 淡棕：皮革、木質、歷史感
- 淡黃/淡粉：郵戳、貼紙、可愛小物收藏

所有顏色、字體、背景、陰影、角落與裝飾，都要透過 theme tokens 管理，
不可散落硬編碼在元件本體。

### 3.4 手帳風格

> **接近旅行手帳**

特徵：

- 細線條
- 紙張質感
- 淡彩色調
- 親手整理的手寫感標籤
- 可愛但不幼稚
- 版面整齊，不像 scrapbook

## 4. 第一階段的四個入口

分類不一樣大，不突顯任何一項。

1. **城堡冒險**
2. **交通震撼**
3. **好物研究**
4. **可愛收藏**

### 4.1 入口卡片視覺

採用差異化插畫，而不是實際照片。

建議意象：

- 城堡冒險：天守、石垣、山城、印章
- 交通震撼：火車、地圖、行李、票券
- 好物研究：泡麵、零食、麵包、餐具
- 可愛收藏：貼紙、便條、玩偶、吉祥物、紀念品

### 4.2 分類架構

分類不是分別彼此獨立的網站，而是同一本手帳的不同分節。

**框架**：右側手帳索引標籤。
**手機**：底部固定分類列，保持核心內容易操作。

利用共用同一組結構的技法，只改變外觀位置。

### 4.3 分類的差異

> **共用手帳骨架，但各自保留小差異。**

- 城堡冒險：地圖、印章、旅程紀念感
- 交通震撼：路線、票券、旅程細節
- 好物研究：餐點照片、廠家推薦
- 可愛收藏：貼紙簿、展示格

不可讓每個分類變成完全不同的視覺系統。

## 5. 最新收藏

### 5.1 版位位置

分類入口下方顯示「最新收藏」。

### 5.2 樣式

採用手帳貼紙感：

- 4～6 張
- 照片、手帳、票根、貼紙帶漸層或斜角
- 不可過度花俏
- 進維護時保持排序容易且手機的可用性

### 5.3 每張顯示資訊

只顯示：

- 照片
- 標題

不顯示日期、分類、長篇文字或完整心得。

### 5.4 更新規則

預設自動顯示最近核准的 4～6 張。

Minz 可以：

- 選圖
- 退版
- 調整順序

平常不必特別維護，但保留人工策展能力。

## 6. 技術方案與 Minz 頁面的邊界

這是本文件最重要的架構決策。

### 6.1 技術方案的定位

技術方案負責一整本旅程的完整記錄，包含：

1. **規劃模式**
2. **實際到達使用模式**
3. **技術直記模式**
4. **完成後不修改，成為正式模式**

每一本手帳最後都會從規劃走一路使用到完成結果，不另外複製成另一份
資料。

### 6.2 技術方案負責的內容

- 每日行程
- 預計時間
- 實際走訪時間
- 地點與路線
- 交通
- 每一餐與餐廳
- 住宿與訂位
- 票券與私人資料
- 現場操作
- 行程調整
- 照片
- 技術直記
- 完成後排版
- PDF 輸出

### 6.3 Minz 頁面的定位

Minz 頁面是：

> **個人收藏，內容選擇**

它負責：

- 展示已完成的遊記
- 顯示封面、標題與簡介
- 提供「精選遊記」
- 依需求提供 PDF
- 從遊記中精選封面卡片、好物與可愛收藏
- 呈現 Minz 的個人偏好與觀點的方式

### 6.4 不可發生的事

Minz 頁面不可：

- 取代技術行程管理
- 維護第二份交通、地點、票券資料
- 複製完整遊記資料
- 讓技術方案的另一套排版 Theme
- 讓訪客進入幕後技術工作空間
- 自動公開未完成或未整理的內容

### 6.5 資料來源

> **技術方案是完整資料的唯一來源，Minz 頁面只保存展示設定與來源引用。**

Minz 頁面可以保存：

- `sourceProject`
- `sourceId`
- `sourceChapterId`
- 顯示標題
- 封面選擇
- 所屬分類
- 簡介介紹
- 排序
- 是否置頂
- 公開範圍
- 是否允許搜尋引擎收錄

不可複製來源內容的完整正式結構與技術資料。

## 7. 遊記完成與收藏展示流程

### 7.1 遊記完成之後

遊記不直接自動加入 Minz 首頁。

流程：

1. 技術方案完成遊記提交
2. 系統把遊記登入 Minz 待審核，待展示。
3. 系統自動生成：
   - 建議封面
   - 建議標題
   - 建議簡介
   - 建議分類
4. Minz 可以：
   - 一鍵確認
   - 改封面
   - 改標題
   - 改分類
   - 調整公開範圍
   - 忽略

### 7.2 維護者

> **技術方案負責把系統做好，但不負責 Minz 頁面的日常維護。**

Minz 必須能自己用已完成的資料維護內容，不用寫程式，也不需依賴
Jonathan 幫忙操作。

### 7.3 操作複雜度

預設案：

- **B：簡易調整後發布**
- 同時保留 **A：一鍵發布**

系統會完成大部分整理，Minz 通常只需確認或微調一兩項。

## 8. 遊記內容的分項收藏流程

遊記完成後，系統可辨識其中值得展示的項目，建立分項收藏。

例如：

- 犬山城 → 城堡冒險
- 鳥貴族總本家 → 好物研究
- PIYORIN → 可愛收藏

### 8.1 分項規則

- 分項內容不可自動公開
- 進入待展示區
- Minz 可一鍵核准、修改或忽略
- 必須保留來源遊記與章節關聯
- 不複製完整來源資料

### 8.2 同一來源可多篇分展示

同一本遊記可以出現在：

- 交通震撼：整本遊記
- 城堡冒險：其中的城堡篇章
- 好物研究：其中的餐廳內容
- 可愛收藏：其中的紀念品或紀念章

底層仍然只有一份正式資料。

## 9. 完成內容與遊記瀏覽方式

### 9.1 多層呈現

分類多層顯示輕巧收藏卡，不放大量資訊。

### 9.2 點選方式

點選後翻到新的手帳章頁。

要求：

- 支援獨立網址
- 可直接分享
- 保留手帳視覺一致性
- 不使用中央跳窗放大顯示內容
- 不強制使用左右仿真翻頁

### 9.3 遊記順序依內容類型調整

- 城堡冒險：偏向簡短歷史與實際體驗
- 交通震撼：偏向使用手法或路線
- 好物研究：偏向住家或餐點
- 可愛收藏：偏向收藏亮點

### 9.4 遊記形式

> **網頁瀏覽為主，並可輸出 PDF。**

**網頁版**

- 手機優先
- 上下滑動
- 可放地圖、相簿與互動內容

**PDF**

- 固定排版
- 可收藏、列印、寄給親友
- 跟網頁版共用同一份遊記資料
- 不另外維護第二份內容

## 10. 訪客與授權模式

### 10.1 主要原則

> **親和為主，但網頁可能被陌生人取得，因此公開內容必須能安全面對陌生
> 訪客。**

### 10.2 不要求成為註冊

訪客不需要建立 Jonaminz 帳號。

採用：

- 平常：分享連結
- 較敏感內容：連結＋簡易密碼
- 管理者：Jonathan、Minz 正式登入

### 10.3 內容可見層級

建議資料層級：

```text
public           所有人可見
shared_link      持有專屬連結可見
link_passcode    持連結並輸入密碼可見
owners           僅 Jonathan、Minz 可見
```

`unlisted` 不當作為安全本質，而是視為分享用途、索引藏起。

### 10.4 授權粒度

不預設整個類別內容一定屬於親和層。

授權可設定到：

- 每一本遊記
- 每一篇內容
- 每一張照片
- 必要時每一個內容單位

例如同一篇城堡文章可公開，但其中人物照片只給有連結的親友。

### 10.5 專屬連結要求

專屬分享連結應支援：

- 隨時撤銷
- 可設定期限
- 可限制可見的內容範圍
- 不顯示在公開 sitemap
- 不被搜尋引擎收錄

### 10.6 絕對不可公開的人資訊

- 即時位置
- 未來精確行程
- 完整通訊錄
- 旅行細節
- 私人事物
- 完整的真實姓名
- 工作地點
- 電話
- 私人聯絡資訊
- 住宿訂位資料
- 票券條碼
- 護照代碼

「營養師」身份可公開，但不公開工作單位與可辨識身份。

## 11. 搜尋引擎與公開性

### 11.1 預設規則

所有新頁面預設：

```text
noindex
```

包含：

- Minz 首頁
- 新遊記
- 新收藏
- 新完成內容

### 11.2 個別選擇

每個公開頁面可以由 Minz 個別選擇是否允許搜尋引擎收錄。

### 11.3 永遠不可收錄

以下內容永遠 `noindex`：

- 專屬連結內容
- 連結＋密碼內容
- owners-only 內容
- 後展示草稿
- 未完成遊記
- 私人管理頁面

### 11.4 Sitemap

Sitemap 只收錄：

- 已公開
- 且 Minz 明確允許被收錄

的頁面。

## 12. 後台維護體驗

### 12.1 核心目標

Minz 應該在整理她的相簿與收藏，而不是維護網站。

日常流程：

> **看到新內容 → 確認封面、分類 → 發布**

### 12.2 後展示事件狀態

建議狀態：

```text
suggested
draft
published
ignored
archived
```

### 12.3 每筆收藏可操作項目

- 預覽
- 一鍵發布
- 改封面
- 改標題
- 改簡介
- 選分類
- 設定公開範圍
- 設定是否可被搜尋
- 置頂
- 調整順序
- 忽略

### 12.4 行動裝置優先

Minz 很可能直接用手機維護，因此後台不可只優化桌機方案。

手機優先：

- 大型點擊區
- 少輸入本位
- 快速的預覽
- 不要求擁有設計能力完成核心操作
- 一鍵發布為最優顯
- 隱私變更要確認

## 13. 建議資料模型（概念層）

以下不是最終 DB schema，而是 Agent 實作時必須保留的資料關係。

### 13.1 分類 registry

```json
{
  "id": "castle-adventures",
  "title": "城堡冒險",
  "icon": "castle",
  "order": 10,
  "visible": true,
  "parentId": null,
  "layoutPreset": "notebook-castle",
  "themeVariant": "castle"
}
```

### 13.2 Minz 展示項目

```json
{
  "id": "minz-item-inuyama-castle",
  "sourceProject": "travel",
  "sourceType": "travel-book-chapter",
  "sourceId": "nagoya-inuyama-2026",
  "sourceChapterId": "inuyama-castle",
  "collectionId": "castle-adventures",
  "displayTitle": "犬山城",
  "coverAssetId": "asset-selected-03",
  "shortDescription": "",
  "visibility": "public",
  "searchIndexing": "noindex",
  "status": "published",
  "pinned": false,
  "order": 20
}
```

### 13.3 分享通行證

```json
{
  "id": "share-pass-family-2026",
  "scope": ["travel-book:nagoya-inuyama-2026"],
  "mode": "link",
  "passcodeRequired": false,
  "expiresAt": null,
  "revokedAt": null
}
```

### 13.4 重要資料原則

- 展示項目只引用來源
- 不複製完整遊記內容
- 授權判斷必須在資料模型分層處理
- 前端渲染不等於授權保護
- 分享 token 不可直接使用可猜測 ID
- 所有內容不可只靠 `noindex` 保護

## 15. 驗收標準

### 15.1 定位驗收

- 首頁第一眼像 Minz 的個人生活選集，而不是技術工具後台
- 技術內容突出，但不把 Minz 永久定義成只有技術
- 人物特色存在於細節，不誇張不簡陋

### 15.2 結構驗收

- 分類全部來自 registry
- 調整任一分類不需修改頁面骨架
- 新增分類不需觸碰首頁
- 空分類不顯示
- Minz 展示項目只保存來源引用與展示 metadata
- 不存在第二份完整遊記資料

### 15.3 維護驗收

- Minz 可在手機上獨立完成一類內容的發布
- 正常思考三步：預覽 → 確認或微調 → 發布
- 可一鍵發布
- 可修改標題、簡介、分類
- 可置頂、排序、分類
- 不需 Jonathan 協助日常維護

### 15.4 隱私驗收

- 未授權連結無法轉換為分享連結內容
- 密碼內容不可只依賴前端 CSS/JS 隱藏
- 分享連結可撤銷
- 私人頁面永遠 `noindex`
- Sitemap 不包含未允許收錄內容
- 每筆私人內容單位保留未來獨立技術授權能力

### 15.5 RWD 驗收

- 桌機索引列、手機變成底部固定導覽
- 貼紙在手機改為時間線排序
- 分類卡符合觸控操作
- 不強制使用左右翻頁效果

## 16. Agent 絕對不可誤解的事項

1. **Minz 頁面不是技術方案。**
2. **遊記不是技術結果後重新複製建立的另一份資料。**
3. **同一本遊記從規劃、實際使用、直記到完成不換一份資料。**
4. **Minz 頁面只展示完成作品與個人選擇。**
5. **Minz 自己維護頁面，Jonathan 不負責日常展示。**
6. **系統完成大部分預設，Minz 只需確認或微調。**
7. **分項內容不可自動公開。**
8. **公開不需要註冊帳號。**
9. **網頁可分享代表內容可被搜尋引擎收錄。**
10. **`noindex` 不是安全機制。**
11. **目前四個分類不可永久寫死。**
12. **第一版不可顯示空分類或「敬請期待」。**
13. **不可把頁面做成跟普通人格不符的技術風格。**
14. **不可為了手帳質感或裝飾犧牲操作。**
15. **不可強制使用左右翻頁式電子書效果。**

## 18. 建議實作順序

**Phase 1 — 純展示骨架**：建立 theme tokens、section registry、完成
首頁/分類頁/About/最新收藏，用 mock data 驗證 RWD 與視覺。

**Phase 2 — 策展後台**：建立後展示事件佇列，完成預覽、一鍵發布、微調，
加入排序、置頂、分類，使用假的 travel source reference 驗證資料流。

**Phase 3 — 技術方案整合**：接收遊記提交事件，自動建立後展示遊記，
產生分項（好物、可愛收藏分類），保留來源章節引用。

**Phase 4 — 訪客授權**：專屬分享連結、撤銷與期限、連結＋密碼、owner
管理登入、授權層資料庫。

**Phase 5 — 出版與索引**：遊記公開排版、PDF 輸出、個別 indexing 設定、
sitemap 生成。
