# Contract 核准人工檢查清單

**用途**：在 `/pages/admin/contracts/` 按下「核准」之前，人工逐項確認。
這份清單補的是自動驗證（schema／URL／origin）測不到的部分——「外部專案
實際上線的頁面有沒有真的做到合約宣告的事」。自動驗證擋語法，這裡擋語意。

**背景**（2026-07-14 裁決，見 `AI_CONTEXT/DECISIONS.md` 合約分級條目）：
平台對外部專案的要求分兩級——**功能合約**（要有這個行為，長相自己決定，
如 header 的回主頁連結/登入狀態）與**視覺合約**（連長相都要一致，如
Chat 浮動入口）。視覺合約的合規方式是「使用平台提供的元件」（SDK 自動
掛載的 iframe，同一份程式碼保證長相一致），不是照著規格自己刻一份；
所以檢查的重點是「元件有沒有正常出現、有沒有被宿主頁面蓋住/藏起來」，
不是拿尺量像素。

## 每次核准前（所有專案通用）

- [ ] 實際打開合約 `entries[]` 裡每個 `url`（用 admin/contracts 頁的
      diff 檢視拼出完整網址，或 design 頁的「進入」連結），確認頁面
      真的存在、不是 404/白頁。
- [ ] diff 檢視裡的中文字正常顯示，沒有亂碼（曾發生過 shell 編碼壞掉
      的提交，見 CHANGELOG 2026-07-13/14 travel snapshot #6 事件）。
- [ ] `visualIdentity` 宣告的調性跟實際頁面看起來是同一回事（純目視，
      不用精確比色）。

## 合約宣告了 `chat.launcher@1` 時（視覺合約）

- [ ] 打開該專案的正式頁面，右下角有浮動 Chat 入口（未登入 jonaminz
      時入口是隱形的——用**已登入 jonaminz 的瀏覽器**檢查，或先在
      www.jonaminz.com 登入再開該專案頁面）。
- [ ] 入口沒有被宿主頁面的其他元素蓋住（iframe 的樣式隔離擋得住 CSS
      入侵，擋不住宿主自己疊一個更高 z-index 的東西上去）。
- [ ] 點擊入口正確跳轉到 `https://www.jonaminz.com/pages/chat/`。
- [ ] 宿主頁面自己的功能不受影響（SDK 失敗只能 degraded，不能把人家
      頁面弄壞——S24）。

## 未來新增其他視覺合約 capability 時

比照 `chat.launcher@1` 的段落格式，在這份清單加一節：元件有沒有出現、
有沒有被蓋住/藏起來、互動行為對不對、宿主頁面不受影響。四件事。
